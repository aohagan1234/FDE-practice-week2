**MedFlex Engagement | Discovery: 2026-05-12**

## Workflow delegation map

| Workflow step | Current method | Delegation level | Reason |
|---|---|---|---|
| Shift request intake — receive and parse unstructured request into structured job order | Coordinator reads email/phone/portal manually and enters into CRM | **Agent-led + human oversight** | Requests arrive in arbitrary natural language across three channels. The same shift requirement can be expressed as "ICU nurse, overnight Tuesday" in an email or as a coordinator phone note. A rule cannot reliably extract facility type, shift time, specialism, and credential requirements from variable-format input. Agent parses; coordinator reviews the structured output before processing begins. |
| Candidate eligibility pre-filter — check credential match, portal availability, proximity | Coordinator checks each criterion manually per candidate | **Human-led + automation support** | The three filter criteria are deterministic: credential X required / nurse has credential X (Y/N); nurse availability status on portal (Y/N); nurse within N miles of facility (distance calculation). These are rule-based lookups, not reasoning. A deterministic filter function runs faster and cheaper than an agent. Note: availability data quality is a risk (stale data from manual updates) — this is a data problem, not a reason to make the filter agentic. |
| Candidate ranking — order eligible candidates for coordinator review | No ranking; coordinator decides how many to propose and who, based on experience | **Fully Agentic** | When multiple eligible candidates exist, the right ranking depends on at least six context-dependent factors simultaneously: credential specificity for this facility type, proximity, hospital's stated preferences (if any history), nurse's response rate, nurse's no-show history at this facility, and current workload relative to the pool. No fixed rule can weigh these correctly across all shift types and hospital relationships. The agent reasons over the full candidate profile per job order and outputs a ranked shortlist with a stated reason per candidate. |
| Parallel outreach — contact top-ranked candidates simultaneously | Coordinator contacts one candidate at a time sequentially; waits for response before contacting next | **Agent-led + human oversight** | The decision of which candidates to contact, how many simultaneously, via which channel, and in what order requires reasoning over urgency, pool depth, channel preference, and expected response latency. The execution (sending message via SMS/email) is automated. The strategy is agentic. Coordinator is notified of outreach initiated but does not approve each individual contact. |
| Response tracking — monitor for nurse confirmations within target window | Coordinator monitors manually; follows up if no response | **Human-led + automation support** | If no confirmation is received within the target window, escalate to coordinator. This is a scheduled job: time threshold + status check + trigger notification. No contextual reasoning required. A deterministic workflow handles this more simply and reliably than an agent. |
| Credential validation at placement — verify credential currency before confirming a candidate | Not currently done at placement time; mismatches detected post-shift only | **Agent-led + human oversight** | For clear cases (credential A required / nurse has credential A, last verified 30 days ago) this is deterministic. For ambiguous cases ("ICU-certified preferred" vs nurse has "step-down ICU experience"), the agent reasons over credential detail, facility requirements text, and precedent from prior placements at that facility. The ambiguous case cannot be resolved by a rule. Agent validates; coordinator reviews all partial-match or ambiguity flags before placement proceeds. |
| Coordinator confirmation gate — final approval of agent recommendation before hospital notification | Not applicable (coordinator already owns all decisions) | **Human only** | The final placement decision is the coordinator's. The agent surfaces the recommended match with rationale; the coordinator approves or overrides. This is non-negotiable: Marcus confirmed two prior AI projects failed partly because staff did not trust machine outputs. Coordinator accountability for the placement decision must remain explicit. Removing this gate is a Wave 2 consideration only, conditional on demonstrated agent accuracy. |
| Hospital notification — send confirmation once placement is approved | Coordinator sends confirmation email manually | **Human-led + automation support** | Triggered automatically on coordinator approval. Template + hospital contact + shift details = deterministic workflow. No contextual reasoning required. |
| No-show backfill — if a placed nurse no-shows, find replacement rapidly | Coordinator runs a "fire drill" manually to find replacement | **Agent-led + human oversight** | Backfill is candidate ranking under a time constraint with a reduced pool (some candidates already committed to other shifts). The same factors apply as in standard ranking, but with urgency weighting. A rule cannot adapt to a reduced pool with urgency; an agent can. Coordinator monitors and can intervene at any point. |
| Mismatch detection — identify credential or skill gap before shift begins | Currently only detected post-shift via hospital satisfaction survey | **Human-led + automation support** | Pre-shift detection is a scheduled check: compare placed nurse's credentials against facility's requirement record T-hours before shift. This is deterministic. Flag to coordinator if gap is found. The scheduled job replaces the current post-shift survey as the detection mechanism for a subset of credential mismatches. |

## ADR 1 — Coordinator approval gate at final placement

**Decision:** The agent handles intake parsing, eligibility filtering, candidate ranking, parallel outreach, and credential validation autonomously. The coordinator retains the final placement confirmation decision before the hospital is notified. The coordinator does not approve each intermediate step.

**Alternative A: Fully autonomous placement** — agent confirms placement with hospital directly without coordinator review.
- Consequences: Eliminates the human bottleneck entirely; fastest theoretical path to <1h fill time. However, if the agent makes a credential or judgment error, there is no human catch before a nurse is confirmed and the hospital is notified. Two prior AI projects eroded coordinator and management trust in machine outputs. A fully autonomous system that makes even occasional visible errors will be rejected faster than the previous recommendation engine was. Compliance exposure if a credential gap is confirmed without human verification. This option is not viable in Wave 1.

**Alternative B: Coordinator reviews and approves every intermediate step** — coordinator approves the structured job order, the candidate shortlist, the outreach list, and the final placement.
- Consequences: Maximum human oversight and lowest compliance risk. But reintroduces the coordinator as a bottleneck at every stage. Fill time improvement will be marginal because the coordinator's sequential approval gates recreate the current sequential workflow. The agent adds no capacity benefit if the coordinator must review each step.

**Why the chosen option was selected:** The coordinator approval gate at the final step only removes the coordinator from the time-critical matching and outreach work while preserving human accountability for the actual placement decision. The agent does the parallel, high-volume work; the coordinator makes the commitment. This is the minimum trust requirement given two prior failed AI projects, and it is the right starting point for building coordinator confidence.

**Conditions that would require this decision to be revisited:** If coordinator approval time consistently adds more than 30 minutes to fill time across 30+ shifts, AND if agent first-recommendation accuracy exceeds 90% over 60 operational days, the gate can be relaxed for a defined subset of "clean" shift types (exact credential match, no prior no-show history, nurse confirmed availability on portal within last 2 hours).

## ADR 2 — Real-time credential validation at placement vs. reliance on compliance batch state

**Decision:** The agent triggers a real-time credential status check against the compliance system at placement time for every candidate before they are ranked or outreached. Candidates with credentials last verified more than 90 days ago, or with a lapsed status, are excluded from the shortlist and flagged to the compliance team — but this does not block other candidates from proceeding.

**Alternative A: Trust the compliance system's quarterly-updated credential state without additional checks.**
- Consequences: Simpler implementation. Agent reads whatever is in the system. But the discovery call confirmed that credential re-verification happens within a week of a state regulatory ping — meaning there is a window of up to 7 days (and potentially longer between pings) where a nurse with a lapsed credential remains on the active roster. If the agent relies on batch state without a recency check, it will recommend and potentially place a nurse whose credential lapsed after the last quarterly update. This is the most likely credential-gap contribution to the 7% mismatch rate. The architecture would build in the same failure mode the current process already has.

**Alternative B: Route every placement through the compliance team for credential verification before proceeding.**
- Consequences: Eliminates credential lag entirely at the cost of adding the compliance team to the critical path. If compliance team turnaround is not measured in minutes, this makes the <1h fill time target unachievable for any shift that requires verification. It also increases compliance team workload proportionally to shift volume — the opposite of the "10x without 10x-ing" mandate.

**Why the chosen option was selected:** A recency-gated real-time check (is the credential status current, and was it verified within 90 days?) is deterministic and fast — it does not require compliance team involvement for the common case where credentials are current. It only routes the exception (lapsed or stale credential) to the compliance team. This eliminates the placement window for known-lapsed credentials without creating a human bottleneck for every shift. The 90-day threshold aligns with the quarterly compliance cadence Marcus described.

**Conditions that would require this decision to be revisited:** If Aaron confirms that the compliance system cannot support real-time API queries, the agent falls back to a soft-SLA model: flag the placement as pending compliance confirmation, route to Linda's team with a 2-hour SLA for active placement requests. Fill time target remains achievable for most shifts where credentials are current; lapsed-credential cases accept a longer fill time as the compliance cost.
